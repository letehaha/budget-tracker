1Concepts & data model

Product principle – the balance is a user-maintained principal ledger

A loan's balance moves only on user-recorded events: the initial snapshot, payments, and manual balance corrections. Every recorded payment is 100% principal – linking a $1,000 transaction always reduces the owed amount by exactly $1,000; no principal/interest split exists anywhere in the schema. The app never simulates lender-side interest accrual onto the balance. Abolished alternative (do not reach for it): auto-accruing interest from APR between payments – rejected because lender math varies too much across day-count conventions, posting rules, fees, and jurisdictions to reproduce; a confidently-wrong balance the user must fight is worse than an honestly-simple one the user understands. APR is used exclusively for forward projection and for estimating interest; it never mutates the balance.

Consequences, all intended: the ledger understates the lender's figure by accrued interest, so it can reach zero – and settle the loan – before the lender's books do (e.g. 10 × $1,000 payments clear a $10,000 ledger while the real loan still carries the interest portion). Reconciliation via balance corrections (§4) is the designed workflow, not a workaround: the user restates "I actually still owe $X" from a lender statement, a prematurely-settled loan reopens (§7), and tracking continues. Every interest number in the UI is an amortization-schedule estimate that must be labeled "(est.)". This lens applies to every projection and cost figure in §7–§9.

What a loan is

✅ SPEC A loan is an Account with accountCategory: 'loan', type: 'system', plus a required 1:1 LoanDetails sidecar row carrying loan-only metadata (APR, term, original principal, payment plan, event log, balance anchor). No API path creates a loan-category account without its sidecar in the same DB transaction; UNIQUE(accountId) enforces at most one sidecar. Loan accounts still appear in GET /accounts; the /loans resource is additive.

Balance sign convention

✅ SPEC Account.currentBalance stores the outstanding amount negative (liability convention) so net-worth aggregation picks loans up automatically. The API accepts positive "outstanding" decimals on input (create initialBalance, edit currentBalance) and the service negates them. API responses return the signed (negative) balance. An active loan therefore always has currentBalance < 0; exactly 0 means settled. A positive (credit) balance is unreachable: recomputeLoanBalance floors currentBalance/refCurrentBalance and every rebuilt Balances row at zero (zero-cap rule, §6), and the frontend renders outstanding amounts via max(0, −balance) (outstanding-amount.ts) so a positive value could never display as debt.

✅ SPEC Example: a mortgage originated 2020-06-15 (startDate) is added to the app on 2026-07-01 with $245,100 still owed. The anchor is stamped 2026-07-01 and initialBalance = −245,100. Payments

✅ SPEC A payment is an ordinary transfer pair with transferNature: 'transfer_to_loan': an expense leg on the source account and an income leg on the loan account, linked by transferId. The loan account only ever holds income legs. Income legs move the negative balance toward zero.

Balance maintenance & history

✅ SPEC Loans opt out of the incremental per-transaction balance cascade used by ordinary accounts. Instead, recomputeLoanBalance derives the authoritative balance from scratch on every payment write/edit/delete and every re-anchor: currentBalance = initialBalance + Σ counted income legs (same for ref amounts). It also rebuilds the loan's Balances history rows from the anchor day forward – one row on the anchor date plus one cumulative row per later day with legs; pre-anchor rows stay untouched. Rebuilt rows are floored at zero (zero-cap rule, §6). Balances.amount stores the base-currency (ref) running balance; the balance-journey chart does not read it – it consumes the loan-currency series from GET /loans/:id/balance-history (§7).

2Loan creation

✅ SPEC POST /loans creates the Account + LoanDetails in one DB transaction. The create dialog collects all fields; on success the app navigates to /loans/:id. The loan's currency is added to the user's currency list if new. The events log starts empty. All ref* fields are computed at the creation-time exchange rate and frozen.

Edge cases

  • ✅ SPEC Balance > principal: a legitimate state (negative amortization / capitalized interest). Allowed with a non-blocking inline warning (forms.loan.balanceExceedsPrincipalWarning) – no confirm step, no submit block, no upper cap. Projection floors "principal paid" at 0 rather than showing negative progress.
  • ✅ SPEC Anchor stamping: balanceAnchorDate is always stamped today (server date) at creation – the entered outstanding is true as-of the moment of entry, regardless of startDate.
  • ✅ SPEC Date-ordering invariants: origination ≤ today, and the balance anchor cannot predate origination

    Three rules keep the loan's dates coherent:

    • (a) startDate > today is rejected with 422 on both create and edit – a loan cannot originate in the future. Enforced in the request schema (loans/zod-schemas.ts) via a shared notFutureCalendarDate() check that also gates currentBalanceAsOf. The ceiling is server-tomorrow so a client one calendar day ahead of UTC isn't rejected for its own valid local date.
    • (b) A balance correction whose effective as-of (currentBalanceAsOf ?? today) is earlier than the effective start date (startDate from the same PATCH, else the persisted value) is rejected with 422 – an outstanding balance cannot be asserted for a date before the loan existed. This is a cross-field rule, so it lives in update-loan.service.ts (the schema can't see the persisted startDate when as-of arrives without one) and runs before any DB write.
    • (c) Together (a) and (b) guarantee balanceAnchorDate ≥ startDate: at creation the anchor is today and origination is ≤ today; a correction can only move the anchor to a date ≥ origination.

    The client mirrors both rules in the pickers: the origination field caps at today, and the balance-as-of field's minimum is startDate.

  • ✅ SPEC Missing/invalid required field → 422 with field-level Zod details. Huge principals compute exactly – all arithmetic is integer cents.

3Loan editing

✅ SPEC PATCH /loans/:id takes any subset of fields (at least one required, else 422). The service diffs each field against the persisted row; timeline events are appended only on a real change – echoing the current value is a no-op and must not create an event. Edits run in one DB transaction. 404 when the loan isn't owned by the caller.

✅ SPEC startDate is a display-only origination date

startDate records when the loan was issued in the real world. It affects no balance math – it drives only history display (balance-journey chart start, "closed N months early"). Its identity is distinct from the anchor and must read that way in the UI:

  • UI label: "Origination date", with helper text along the lines of "when the loan was issued – used for history display only, doesn't affect balance". It is not an anchor: balanceAnchorDate is the recompute boundary, stays hidden/auto (stamped at creation, moved only by balance corrections §4), and is never directly editable.
  • Edit is silent: no timeline event, no anchor move, no recompute – it is a typo-fix affordance.
  • Validation: startDate ≤ today on create and edit (see date-ordering invariants). No constraint against the anchor or earliest payment date: a paid-off loan imported years after origination legitimately has a startDate long before both.

4Balance corrections

✅ SPEC A balance correction asserts: "the outstanding was currentBalance (positive decimal) as-of currentBalanceAsOf, before that day's payments." It exists so users can reconcile the principal-only tracked balance against lender statements (interest accrual, fees). Mechanics, in order, inside one transaction:

  1. balanceAnchorDate ← asOf (defaults to today when omitted);
  2. Account.initialBalance ← −currentBalance (+ ref at today's rate) written directly – deliberately bypassing the generic account-update diff logic, which would shift instead of re-anchor;
  3. a balance_correction event is appended (from = old outstanding, to = new) – manual edits leave no transaction trail, so this event is the only record;
  4. recomputeLoanBalance re-derives currentBalance = snapshot + post-anchor legs, and rebuilds Balances history from the anchor forward.

Edge cases

  • ✅ SPEC Echo without as-of: sending the current balance unchanged and no currentBalanceAsOf is dropped – no event, anchor stays. Sending the same amount with an as-of date is a genuine re-anchor (the date changes which legs count).
  • ✅ SPEC currentBalanceAsOf without currentBalance: 422 – the anchor date is meaningless alone.
  • ✅ SPEC Future as-of: 422 "currentBalanceAsOf must not be in the future". Ceiling is server-"tomorrow" to absorb client timezone skew.
  • ✅ SPEC Same-day double-count trade-off: correcting to a post-payment statement balance on the very day that payment is logged applies the payment twice (anchor boundary is inclusive). Documented, accepted; users should enter the pre-payment statement figure.
  • As-of before startDate: rejected with 422 – a balance cannot predate origination (date-ordering invariants).
  • ✅ SPEC Overpay guard: a correction is rejected with 422 when the re-anchored snapshot plus the payment legs dated on/after the new as-of date would project a positive (credit) balance – e.g. correcting to $100 as-of two months ago with a $500 later payment (projected +$400). The message names the overpay amount and the total of the offending payments. Projected exactly zero is allowed (a legit backdated payoff). Checked before any DB write, so a rejected correction leaves the anchor, balance, and events untouched. A positive projection is always user error: legitimate statement-sync corrections assert a higher outstanding (accrued interest), never one below the payments already logged since that date.

5Recording payments

✅ SPEC The loan detail page's "Record payment" opens a payment dialog: source account, amount (in source currency), date, note. Submit creates the transfer_to_loan pair; the loan-side income leg amount is in the loan's currency. The recent payments card and the balance update immediately (recompute runs in the same transaction). Creating a generic transfer in the transactions UI with a loan destination switches the nature to transfer_to_loan automatically.

Source account restriction

✅ SPEC The source picker offers system (manually-managed) accounts only, excluding vehicle and loan categories (loans are never payment sources – money only flows in). Bank-synced accounts are excluded because their transactions arrive from the provider; the synced expense should instead be attached via Link payments (§6). The dialog shows a hint under the source picker pointing users to Link payments so the omission isn't silent.

Overpay guard

✅ SPEC Institutional loans cannot go into credit. The server guard (assertLoanPaymentAllowed) row-locks the loan account (SELECT … FOR UPDATE) and rejects any write whose projected balance would be > 0 cents, with 422 transactions.loanPaymentOverpay. The lock closes the two-concurrent-submits race. Applies to create, amount edit (including amount-only PATCHes), date moves, and re-pointing a transfer at a loan. Exact payoff to $0 is allowed (projected balance of exactly zero passes). The client mirrors the guard live with a "max payable: $X" hint (getMaxLoanPayment / isLoanOverpayment, half-cent float slack); the server remains authoritative. The guard applies uniformly to single payments, amount edits, and date moves, rejecting each with a 422.

Pre-anchor payment exemption

✅ SPEC A payment dated before balanceAnchorDate is accepted but does not move the balance, and it skips the overpay guard entirely. Why this exists: pre-anchor payments are already baked into the anchor snapshot (§1) – they are backfilled history, useful for the payments list and spending reports, and mathematically incapable of overpaying because the recompute never counts them. A $300 backfill against a $100 remaining balance is therefore legal. Guard symmetry on edits: when an existing leg is edited, its old amount is credited back only if that leg was itself counted (dated on/after the anchor); moving a leg across the anchor boundary in either direction re-evaluates which side it counts on.

Cross-currency payments

✅ SPEC The source amount is converted to the loan's currency at the payment's own date; validation and storage of the income leg are in loan currency. Missing exchange rate → explicit 422, never a 500.

Editing / deleting a payment

  • ✅ SPEC Editing amount or date, or deleting either leg, triggers the recompute; the guard re-runs on any change that could raise the counted sum. Deleting a payment moves the balance away from zero and can reopen a settled loan (§7).
  • ✅ SPEC Live overpay validation on edit: create and edit share one payment form component, so the edit dialog shows the same live "max payable: $X" hint and surfaces overpay violations before submit exactly like the create dialog.
  • ✅ SPEC Amount-only PATCH keeps the pair consistent: a generic PATCH {amount} on a transfer's source leg re-derives the destination leg 1:1 when the pair shares a currency, and is rejected with 422 (explicit error) when the pair is cross-currency – an API caller cannot desync the legs. Loan overpay validation runs on amount-only edits too.
  • ✅ SPEC Deletes serialize on the row lock: all balance-mutating payment writes, including deletes, serialize on the loan-account row lock – the recompute itself acquires SELECT … FOR UPDATE, so no write path can skip it.
  • ✅ SPEC One day definition: "the leg's calendar day" is the UTC calendar day everywhere. Shared anchor-day helpers (utcDayKey / isOnOrAfterAnchorDay) match SQL DATE(time) in the UTC database and are used by both the recompute grouping and the overpay guard.

Unlinking a payment

✅ SPEC Unlink is a loan-specific endpoint that accepts either leg's id, deletes the loan-side income leg (firing the recompute), and restores the source leg to a plain expense (not_transfer, transferId cleared) – the spend stays on the books. The generic transfer-unlink endpoint rejects transfer_to_loan pairs with 422 – loan pairs are asymmetric (destination leg must be destroyed, not orphaned), so only the loan endpoint may take them apart. Deleting the whole payment (both legs) is the other supported undo. Stale/foreign selections (wrong loan, non-payment transfer) → 422 loans.unlinkPaymentInvalidSelection.

Other rejected inputs

✅ SPEC Zero-amount payments, self-pay (source == loan), and non-loan destinations for transfer_to_loan → 422 with field-level errors.

6Linking existing expenses (bulk)

✅ SPEC POST /loans/:id/link-payments converts existing expense transactions (typically imported from synced accounts) into loan payments in bulk: each becomes the source leg of a transfer_to_loan pair with a generated income leg on the loan account – the same shape as a recorded payment.

Eligibility & atomicity

  • ✅ SPEC 1–500 unique transaction ids per batch (duplicates → 422).
  • ✅ SPEC Every transaction must be: owned by the caller, not_transfer, an expense, and not on the loan account itself. Any ineligible or missing row fails the whole batch with 422 loans.linkPaymentsInvalidSelection – never a partial link.
  • ✅ SPEC Each expense converts to loan currency at its own date's rate; legs dated before the anchor link fine but don't count toward the balance (§5 exemption).

Aggregate overpay check & confirm flow

✅ SPEC Overpay is validated once for the whole batch against the row-locked balance (per-leg guards are skipped – each linked leg moves the balance toward zero and would falsely trip). When the counted sum overshoots: 422 with code: loanPaymentOverpayConfirmationRequired and details { projectedBalance, maxLinkable, overpayBy }; the dialog shows a confirmation. Re-sending with confirmOverpay: true proceeds. The confirm affordance exists because FX rounding can land a batch a few cents over an exact payoff.

✅ SPEC A confirmed overpay caps the balance at zero and pays the loan off

A loan never carries a positive (credit) balance. When a confirmed bulk-link batch overshoots the owed amount, the outstanding balance floors at $0 and the recompute stamps paid_off on the owing → settled transition (§7). The excess is never credited back to the loan – it stays only as the real expense legs on the user's cash account, who owns the selection. The confirm dialog acknowledges this: it warns the payments exceed the remaining balance and that the extra won't be credited back.

The floor lives in recompute-loan-balance.service.ts and clamps both the current balance and every rebuilt Balances history row at zero, so credit is impossible regardless of entry point; a settled loan's last history row sits at 0. Scope is the bulk-link confirm path – the balance-correction path hard-rejects any positive projection and cannot reach a credit state. As defense in depth, the frontend renders outstanding amounts via max(0, −balance) (outstanding-amount.ts), so even an anomalous positive value would display as settled ($0), never as debt.

7Payoff & paid-off state

Zero-crossing detection

✅ SPEC A loan is settled when currentBalance == 0 (projection reports isPaidOff: true for balance ≤ 0). Settlement is ledger-driven (§1): the principal ledger reaching zero settles the loan even while the lender's books still carry accrued interest – that is intended, and the user restates the real remaining amount via a balance correction (§4), which reopens the loan. The recompute stamps a paid_off timeline event exactly on the owing → settled transition (old balance negative, new balance non-negative) – repeated recomputes of an already-settled loan never duplicate it, so at most one live paid_off event exists at any time. The event's at is the date of the payment that settled the loan (a cumulative walk over the post-anchor legs finds the leg that reached zero), not the wall-clock recompute time – a backfilled final payment gets its historical close date; a payoff produced by a correction-to-zero uses balanceAnchorDate. The account stays active; no auto-archive.

Settled presentation

  • ✅ SPEC Detail page shows a settled banner (loan ran X years Y months; closed N months early when termMonths is set – from startDate + termMonths vs close date). List page moves the loan to a separate "Paid off" section.
  • ✅ SPEC Cost breakdown card: total cost = principal + estimated interest (est. only, per §1); split percentages; "interest per $1 borrowed" pill. When interest can't be estimated (no term), the total collapses to principal only. The interest figure is the actual-open-duration estimate (paid-off interest estimate, §8).
  • ✅ SPEC Balance journey chart: plots the outstanding balance in the loan's currency from the opening tracked balance to $0, fed by GET /loans/:id/balance-history. The endpoint returns the outstanding-balance series in the loan's native currency as signed decimals (negative while outstanding, floored at zero), computed from initialBalance plus post-anchor legs grouped by UTC calendar day (matching SQL DATE(time)); legs dated on the anchor day fold into the anchor row. 404 for a missing, foreign, or non-loan account. Months before the first history record fall back to the opening tracked balance (initialBalance), not originalPrincipal. Axis and tooltip are in loan currency; the amortization-schedule overlay derived from originalPrincipal is unit-consistent with the series.
  • ✅ SPEC Payments on a settled loan: the overpay guard rejects any counted (post-anchor) payment – projected balance would exceed 0. Pre-anchor backfill payments remain recordable for history correction (§5 exemption). Balance corrections stay available to restate the loan as still-owing.
  • ✅ SPEC Reopen semantics: editing or deleting a payment that drops the balance below zero resurrects the loan: settled banner clears, it returns to the active list, projection resumes. A reopen removes all paid_off events from the timeline – an active loan never shows "paid off" – and a later re-payoff stamps a fresh one.

8Projections & math

✅ SPEC computeLoanProjection is a pure function (backend common/loans/projection.ts), recomputed on every GET /loans[/:id] – no caching. All arithmetic in integer cents; API outputs are decimals. today is injected. Every interest output is an estimate (§1 principle).

Contract

Input Meaning
currentBalanceCents abs of the signed balance (never positive – the recompute floors at zero, §1). 0 → paid-off branch.
originalPrincipalCents immutable lender principal.
interestRate APR percent, [0, 100).
plannedPaymentCents monthly payment; 0 normalizes to null ("no plan", not "underfunded").
termMonths only used to derive estimatedInterestPaid; null → estimate null.

Algorithm

monthlyInterest = roundHalfToEven(balance × rate / 100 / 12)        // banker's rounding, cents

if balance ≤ 0            → isPaidOff, monthsRemaining 0, totalInterestRemaining 0
if payment == null        → warning 'no_planned_payment'   (payoff fields null)
if payment ≤ monthlyInterest → warning 'payment_below_interest' (payoff fields null)

monthsRemaining = ⌈ ln(P / (P − B·r)) / ln(1 + r) ⌉   where r = rate/100/12    // closed form
                = ⌈ B / P ⌉ when rate == 0
if monthsRemaining > 1200 → presented as 'payment_below_interest'              // 100-yr horizon cap

totalInterestRemaining = Σ per-month accrual (final month pays only the leftover
                         balance + its interest – closed-form P×n − B would overstate)

payoffDate = today + monthsRemaining     // month-end clamped: Jan 31 +1mo → Feb 28/29
paidToDate = max(0, principal − balance);  percent clamped 0..100, 1 decimal
scheduledLifetimeInterest = per-month accrual of (principal amortized over termMonths at rate)
estimatedInterestPaid (active)   = max(0, scheduledLifetimeInterest − totalInterestRemaining)
estimatedInterestPaid (paid-off) = per-month accrual over the first min(actualOpenMonths, termMonths)
                                   months of the original schedule                       // see paid-off note
                                   where actualOpenMonths = whole UTC calendar months, startDate → settle date
                                   (partial trailing month counts as full; settle == start → 0;
                                    settle date = the live paid_off event's at, else balanceAnchorDate)
  • ✅ SPEC Warnings: no_planned_payment (prompt to set one) and payment_below_interest (payment never amortizes, incl. the >1200-month case). UI renders banners with no bogus dates/NaN.
  • ✅ SPEC FE lockstep: the frontend's computePayoffScenario (payoff chart, what-if input) must match the backend to the cent – same closed form, same per-month banker's-rounded accrual, same 1200-month cap, same zero-rate and termination handling.
  • ✅ SPEC estimatedInterestPaid semantics: derived, never measured – payments carry no interest split (§1), so the original amortization schedule (originalPrincipal, termMonths, APR) is the only interest model. Active: schedule-lifetime minus remaining, clamped at 0 (a payoff trajectory faster than the contractual schedule clamps to 0 rather than going negative); null when remaining interest is null. Paid-off: interest accrued over the loan's actual open duration – the schedule's per-month accrual summed for the first min(actualOpenMonths, termMonths) months, where actualOpenMonths counts whole UTC calendar months from startDate to the settle date (a partial trailing month counts as full; settle == start → 0). The settle date is the loan's single live paid_off event's at (§7), with balanceAnchorDate as fallback; when neither is derivable, the estimate falls back to the full lifetime figure. Not a proration: interest is front-loaded, so scaling the lifetime figure by elapsed/term would understate the early months' share. The estimate never reads payment amounts, only dates – a single lump-sum payoff and twelve monthly payments closing on the same date estimate identically. A loan that runs to term yields the full schedule figure; a loan created already at $0 yields ≈ 0. Null whenever term is missing/degenerate.
  • ✅ SPEC Paid-off interest estimate reflects the actual open duration

    The paid-off figure runs the original amortization schedule over the months the loan was actually open (startDate → settle date, capped at termMonths) and sums only that interest – the formula in the semantics entry above; it is the cost-breakdown card's number (§7). Early payoff shrinks the estimate to the front-loaded months actually elapsed; running to term reproduces the full lifetime figure – no special case. A loan created at zero balance settles on its anchor date and estimates ≈ 0.

  • ✅ SPEC Min-payment estimate (form helper): fixed payment that fully amortizes the current outstanding balance over the remaining term – B·r / (1 − (1+r)^−m) where m = termMonths − monthsElapsed(startDate), clamped to ≥1. Interest accrues on the current balance, so this answers "what payment reaches zero by the end of the term from here", which stays correct for over/underpaid and mid-life-created loans. On an on-schedule loan it equals the contractual level payment. Falls back to original principal when no balance is entered yet.

✅ SPEC Repaid-% values both sides at one FX epoch (live rates)

Numerator and denominator are built from loan-currency figures (originalPrincipal, currentBalance), each converted to base at live rates – one consistent FX epoch. The line is hidden when any needed rate is unavailable, mirroring "Interest projected". Abolished alternative: the precomputed ref* fields mix epochs – refOriginalPrincipal is frozen at creation-time FX while refCurrentBalance floats at current FX, so currency moves fake (or hide) repayment progress on foreign-currency loans.

✅ SPEC Repaid-% counts active loans only

Both numerator and denominator sum over active loans only – the % reads as progress on debts the user still carries, consistent with the card's other metrics (monthly obligation, APR, debt-free date). Abolished alternative: including settled loans' principal ("lifetime progress") pads the % with debt that no longer exists – ten old paid-off loans plus one fresh loan would read ~91% repaid. With no active loans the line is hidden entirely rather than showing "100% of $0 repaid".

10Lifecycle: archive / exclude-from-stats / delete

Archive

✅ SPEC Archive sets Account.status = 'archived' via a confirm dialog with an "Also exclude from statistics" checkbox (default on). Archived loans disappear from the sidebar account list and from transaction account pickers (de-emphasized/archived semantics of the shared account store). Unarchive restores active status.

✅ SPEC Archive means "stopped tracking – hide, keep history" (refinanced elsewhere, written off, handed over, tracking abandoned); it is orthogonal to payment state and is the only exit for a loan with recorded payments, which deletion refuses. On the /loans page, archived loans collapse into a collapsed-by-default "Archived" section below "Paid off" and leave the aggregate card and the Active/Paid off lists entirely – aggregates read as "debts the user tracks". Archived wins over paid-off: a settled archived loan appears only in the Archived section. Rows there carry an inline Restore action; the loan detail page shows an archived banner with a Restore button.

✅ SPEC Restore always turns excludeFromStats off: archive auto-enables it (dialog checkbox, default on), and a loan whose tracking resumes should count in stats again. The flag carries no set-by-archive-vs-user provenance, so a manually-set flag is cleared too – the user re-enables it from the actions menu if the loan should stay out of stats.

Exclude from stats

✅ SPEC excludeFromStats: true removes the loan from net worth and dashboard analytics only. It does NOT hide the loan from the /loans list, detail page, or the loans page's own aggregate card – the loans page is a dedicated ledger, not a statistic. Toggle lives in the loan actions menu; failed saves roll the switch back.

Delete

  • ✅ SPEC DELETE /loans/:id → 204. Cascade: account delete runs the shared account-deletion path (share cleanup, notification fan-out); LoanDetails dies via ON DELETE CASCADE.
  • ✅ SPEC Hard block while payment legs exist: any transfer_to_loan leg on the account → 422 loans.deleteBlockedByPayments (payments are real ledger entries; deleting the loan would orphan them). The frontend pre-counts payments (countLoanPayments) to warn before confirm. Timeline events (corrections, notes) deliberately do NOT block – they are self-contained metadata.
  • ✅ SPEC Source-account delete block: deleting an ordinary account that holds loan-payment source legs is blocked with an error naming the loans and payment counts – clear or reassign the payments first.
  • ✅ SPEC Deterministic response: a successful delete always returns 204 – no other success shape exists server-side. Two hardenings keep transport noise from masquerading as application results: the frontend API client treats an empty body as success only when the HTTP status is OK, and the backend drains in-flight connections on shutdown (SIGINT/SIGTERM/SIGUSR2, 5-second cap) so a restart finishes running requests instead of dropping them.

✅ SPEC LoanDetails.events is an append-only JSONB array of typed entries; new event types extend the union + Zod without a migration. at is ISO 8601. Money values are cents at rest, decimals on the wire. Auto-stamped types are never accepted from API input. Timeline renders newest-first with per-type icons and localized copy – no raw JSON, no "undefined".

12Out of scope / future

  • HELOC / lines of credit / multi-disbursement: enum values exist (heloc, business, medical, other) but are hidden from the create picker – the model has no multi-disbursement or redraw support.
  • Refinance: no dedicated flow. Manual path: correct old loan to zero, archive it, create the new loan. The replacedByLoanId FK and refinanced event are reserved schema/type headroom, unpopulated.
  • Currency switch on an existing loan: unsupported, by design (§3).
  • Interest split tracking / auto-split of payments: deferred – the estimate-only principle (§1) holds until a typed interest event or auto-split ships. No system "Loan Interest" category.
  • Payment reminders wiring: paymentDayOfMonth stored, not connected to the reminders engine.
  • Pagination / sorting on GET /loans: the endpoint returns the full list unpaged – deferred while loan counts stay in the single digits.
  • What-if simulator, snowball/avalanche, sharing, MCP tools, liabilities dashboard widget, refund-from-lender event: all deferred.
