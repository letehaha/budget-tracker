CSV / Tabular Transaction Import

How the backend imports transactions from tabular files (generic CSV, BudgetBakers Wallet, YNAB).

Three tabular importers — generic CSV, BudgetBakers Wallet, and YNAB — share one pipeline. Each turns an uploaded file into a list of one intermediate type (ParsedTransactionRow) and writes it through the same transaction service. The parts that are identical across providers live in a shared core/; the parts that genuinely differ (transfer pairing, category hierarchy, balance targeting, column mapping) stay in per-provider folders. Adding a new importer is a parse function + a thin route file that composes core helpers.

1. Overview

Every tabular importer converges on one shared intermediate type — ParsedTransactionRow — and writes those rows as transactions through the same transaction service. Between upload and write there are up to five stages. Not every provider exposes every stage: only generic CSV has the user-driven column-mapping stage (its column layout is unknown), and only CSV and BudgetBakers Wallet run detect-duplicates (YNAB has no dedup stage). The execute stage runs in a background queue for all three providers.

2.1 The shared contract

The single type every tabular importer converges on lives in packages/shared/src/types/import-export.ts: 

interface ParsedTransactionRow {
  rowIndex: number;
  date: string;          // ISO 8601 instant, already anchored to the user's timezone
  amount: Cents;         // integer cents (Money convention)
  description: string;
  payeeName?: string;    // raw value from the mapped Payee column
  categoryName?: string;
  tagNames?: string[];   // split from the source tag cell; absent if no tag column
  accountName: string;
  currencyCode: string;
  transactionType: 'income' | 'expense';
}

Duplicate detection returns DetectDuplicatesResponse (validRows, invalidRows, duplicates, optional dateColumnError and unpriceableRows). Execute takes the re-parseable input (fileContent + the chosen mappings + the skipDuplicateIndices / skipUnpriceableIndices the user confirmed) and returns { jobId }. amount stays integer cents across the whole pipeline; it is reconstructed with Money.fromCents only at write time.

2.2 Stage 1 — Parse csv-import/csv-parser.service.ts

Each provider owns its parser; the generic CSV parser is the most general:

  • Delimiter detection. detectDelimiter tries , ; \t | against the first ~16 KB / first 10 lines and scores each by average column count × consistency, requiring more than one column. Defaults to ,. A delimiter can also be passed explicitly to skip detection.
  • Header safety. The first row becomes headers; empty files and empty header rows throw ValidationError. Prototype-pollution-shaped headers (CSV_FORBIDDEN_HEADERS, e.g. __proto__, constructor) are rejected because rows are later indexed into a plain object by header.
  • Bounds. Data rows are capped at MAX_CSV_ROWS (50 000) to bound downstream mapping and dedup work.
  • Output. CSV's /parse returns { headers, preview (first 50 rows), detectedDelimiter, totalRows } — enough for the UI to render the mapping step. Wallet and YNAB /parse return their full parsed result (their layout is fixed, so there is no mapping step).

2.3 Stage 2 — Column mapping / extract unique values CSV only csv-import/extract-unique-values/index.ts

This stage exists only for generic CSV, because only generic CSV has an unknown column layout the user must map. It re-parses the full file, then runs a validation + extraction fan-out:

  • validateBasicFields, validateCurrency (returns currency column index, default currency, and a currencyMismatchWarning), validateTransactionType, validateExistingAccount, validateExistingCategory — each a small file under the same folder.
  • extractAccounts, extractCategories, extractTags produce the distinct source values the user maps in the UI (link to existing, create new, or skip).

The result drives the mapping UI; the chosen mapping then rides along into stages 3 and 4.

2.4 Stage 3 — Detect duplicates CSV + Wallet csv-import/detect-duplicates/index.ts

This is where raw cells become ParsedTransactionRow and where review data is computed. The shared field parsers and duplicate matcher live in core/; the provider supplies a small bridge:

  • Field parsing. core/parse/parse-amount.ts normalizes US / EU / accounting number formats to cents; core/parse/date-engine.ts (detectDateColumnFormat + parseImportDate) infers a single safe day/month order for the whole column; core/parse/anchor-import-date.ts anchors date-only and zone-less values to the user's IANA timezone (UTC fallback); core/parse/split-tag-cell.ts splits the tag cell.
  • Fail-fast validation. A mapped-but-unreachable Payee column (typo / case mismatch → index -1) throws up front instead of silently importing with no payee data.
  • Duplicate matching. core/duplicates/find-duplicates.ts compares parsed rows against existing transactions and returns DuplicateMatch[] with matchType (originalId / exact / fuzzy) and a confidence score (query capped at 10 000 candidates).
  • Unpriceable rows. core/duplicates/find-unpriceable-rows.ts flags rows whose currency has no stored exchange rate and is neither USD nor the user's base currency, so the review UI can offer skip/abort.
  • Mixed-order guard. When the date column has no single safe day/month order, the response carries a dateColumnError and validRows/duplicates come back empty — the import is blocked rather than guessed.

CSV's detect builds validRows via csv-import/parse-valid-rows.ts (the deterministic parse path the execute worker re-runs). BudgetBakers Wallet runs the same core/duplicates matcher from a ~5-line bridge. YNAB has no detect stage.

2.5 Stage 4 — Execute queued · all providers csv-import/execute-import/index.ts

Execute is asynchronous for every provider: the controller validates the request, enqueues a BullMQ job, and returns { jobId }. The worker re-parses the file server-side from fileContent + the mapping config (parsing is deterministic, so the user's skipDuplicateIndices still line up against the fresh parse), then runs the write loop. The loop runs outside a wrapping transaction: each row's createTransaction opens its own transaction, so a single bad row rolls back only itself and is reported in summary.errors — the rows around it still commit. The contract the user sees is best-effort partial success. Order of operations:

  1. Merge skipDuplicateIndices + skipUnpriceableIndices into one skip set; filter rows; count skipped vs. skipped-unpriceable without double-counting.
  2. addUserCurrencies for every distinct currency in the rows.
  3. core/resolve/create-accounts-if-needed.ts — link-existing verifies ownership; create-new builds a general/system account with zero balances and a refInitialBalance from an injected currency + FX-date callback (CSV uses new Date(); Wallet/YNAB use the earliest parsed date). YNAB passes alwaysCreate: true (no link-existing path).
  4. core/resolve/create-categories-if-needed.ts (flat for CSV/Wallet; createTwoLevelCategoriesIfNeeded for YNAB's Group → leaf) and — if a tag column / labels exist — core/resolve/create-tags-if-needed.ts (mapping mode for CSV; createNamedTagsIfNeeded list mode for Wallet/YNAB). All find-or-create via resolve-or-create-by-name.ts, case-insensitive; only genuine inserts count toward the created totals. Tag-creation failures are fail-fast (see §7).
  5. Per-row write loop. Resolve account + category, build TransactionImportDetails (batchId, importedAt, source), resolve tag ids, then createTransaction with Money.fromCents(row.amount), rawMerchantName from the payee, and the resolved tagIds, ticking onProgress(processed, total) per committed row. When the row supplied tags, payee default tags are re-applied additively so imported tags and payee defaults form a union.

The job result is a summary (imported / skipped / created counts + per-row errors, newTransactionIds, batchId), surfaced over SSE and the status endpoint.

2.6 Stage 5 — Queue & progress all providers core/queue/create-import-job-queue.ts

Each provider has one BullMQ queue bundle built by createImportJobQueue, a shared scaffold parameterized by the few bits that differ (base name, SSE event type, log label, the processJob body):

  • Worker lifetime. The Worker is created at module scope, once per importer. It must never be created inside withTransaction(): Sequelize v7 propagates the active transaction via AsyncLocalStorage, so a Worker built in a transaction context would have its own queries throw "commit has been called".
  • Connection & naming. Its own Redis connection (APPLICATION_REDIS_HOST, maxRetriesPerRequest: null); queue name is namespaced by JEST_WORKER_ID in tests so parallel runs don't cross-fire. Job ids use - (never :, which BullMQ rejects) and a random suffix (not a timestamp) so two imports in the same millisecond don't collide.
  • Retention & retries. attempts: 1 (a user-triggered import must not auto-retry partial commits); completed jobs kept 24h, failed 7d so /status can still fetch them.
  • Progress. The execute service ticks onProgress(processed, total) per committed row. Progress is persisted with job.updateProgress (so /status polling works with no SSE listener) and emitted over SSE; completed/failed always carry the true counts. Progress emission is best-effort — a Redis/SSE hiccup is swallowed and logged, never failing a committed row.
  • Status read race guard. getImportProgress snapshots the job via getJob and then re-queries state; when a job flips to completed/failed in that window before its returnvalue/failedReason has surfaced, it reports running so the next poll picks up the real terminal payload (rather than an empty summary or "Unknown error").

3. The shared core …/import-export/core/

core/ is a flat bag of pure functions and the queue scaffold — no base class, no inheritance. Providers compose these functions; they do not implement a common parse interface.

The provider-local parse-amount.ts / parse-date.ts in budget-bakers-wallet-import/ and ynab-import/ are not duplicates of the core ones — each handles that vendor's specific number/date contract (e.g. YNAB's $1,234.56 + US MM/DD/YYYY; Wallet's ISO instants). The core parse-amount/parse-date serve the generic CSV importer.

4. The three providers

All three converge on ParsedTransactionRow and reuse the same core/ primitives for the back half (field parsing, dedup, resolve, queue). What stays per-provider is the genuinely-divergent logic — the file-format parse and a few structural concerns.

4.1 What is shared vs provider-specific

Stage Shared core Provider-specific
parse core/parse/* field parsers column → ParsedTransactionRow mapping; CSV delimiter detect; Wallet pair-transfers; YNAB Outflow/Inflow + flag + starting-balance
column mapping — CSV only (extract-unique-values/)
detect-duplicates core/duplicates/find-duplicates.ts + find-unpriceable-rows.ts CSV + Wallet: a small accountNameToId bridge. YNAB: no detect stage.
execute queued; worker re-parses, then addUserCurrencies → create-accounts-if-needed → create-categories-if-needed → create-tags-if-needed → per-row createTransaction loop ticking onProgress two-level categories (YNAB); balance targeting (Wallet); transfer writes (Wallet/YNAB)
queue core/queue/create-import-job-queue.ts with default payload builders — one bundle per provider base name, SSE event type, processJob body
HTTP shared createParseController / createStatusController factories (§5) execute & detect controllers + the per-stage Zod schema

Money convention is uniform: ParsedTransactionRow.amount stays integer cents; execute reconstructs via Money.fromCents; no raw: true. Progress payloads carry plain counts, not money.

4.2 Where the divergence is real (and stays provider-owned)

  • BudgetBakers Wallet transfer pairing — pairTransfers() in parse-budget-bakers-wallet.service.ts: ~1% ref-amount tolerance, out-of-wallet leg promotion. No CSV/YNAB equivalent.
  • YNAB structure — two-level categories (Group → leaf), flag-color → tag mapping, Transfer : payee prefix, synthetic starting-balance rows, always-create accounts.
  • BudgetBakers Wallet balance targeting — restores linked-account balances after import (runs after create-accounts-if-needed, inside the Wallet executor).
  • CSV-only specifics — delimiter auto-detection, prototype-pollution header guard, and the entire user-driven column-mapping stage.
  • FX date for opening balance — CSV uses new Date(); Wallet/YNAB use the earliest parsed date. Load-bearing: getting it wrong skews refInitialBalance. Wired explicitly per provider through the create-accounts-if-needed callback.

4.3 AI importers — a separate pipeline

statement-parser/ and investment-transactions-parser/ do not use this column-mapping pipeline: AI extraction instead of a deterministic parse, no extract-unique-values stage, and different skip schemes (statement uses skipIndices: number[], investment uses skipTempIds: string[]). They live alongside but are not part of the tabular core. Their one shared tie-in: investment-transactions-parser/csv-parser.service.ts reuses parseDate from core/parse/parse-date.ts.

5. HTTP surface & controller factories

Each provider keeps its own route file and its own URL namespace; there is no dynamic dispatch route (see §7). The endpoints, by provider:

Provider Endpoints Route file
CSV POST /csv/parse, /csv/extract-unique-values, /csv/detect-duplicates, /csv/execute; GET /csv/execute/status/:jobId routes/import-export/csv.route.ts
BudgetBakers Wallet POST /budget-bakers-wallet/parse, /detect-duplicates, /execute; GET /budget-bakers-wallet/status/:jobId routes/import-export/budget-bakers-wallet.route.ts
YNAB POST /ynab/parse, /ynab/execute; GET /ynab/status/:jobId routes/import-export/ynab.route.ts

All execute/parse routes carry authenticateSession + a rate limiter + validateEndpoint; the status routes carry auth + validation but no rate limiter (the frontend polls them every ~2 s).

The parse and status controllers are identical boilerplate across providers, so they are built by two shared factories in controllers/import-export/helpers/: 

// status — identical except the progress reader + the not-found copy
export const csvStatusController = createStatusController({
  getProgress: getCsvImportProgress,
  notFoundMessage: 'CSV import job not found.',
});

// parse — validate body, run a sync parser, wrap the result
export const parseBudgetBakersWalletController = createParseController({
  schema: fileContentParseSchema,                 // CSV passes its own (it also takes a delimiter)
  parse: ({ fileContent }) => parseBudgetBakersWalletCsv({ fileContent }),
  // wrap defaults to { result }; CSV overrides with identity (it returns the value flat)
});

The execute and detect-duplicates controllers stay per-provider: their request Zod schemas genuinely differ (CSV sends a full column mapping; Wallet/YNAB send their own account-mapping shapes), so a single handler would have to branch on the provider anyway. Factoring only the controllers that are actually identical keeps full per-endpoint static typing and per-route middleware.

6. Adding the Nth importer

To add, say, a "Revolut CSV" importer, the work is:

  1. services/import-export/revolut-import/parse-revolut.service.ts — the only file with new logic. A pure function producing ParsedTransactionRow[] (+ invalid rows). Reuse core/parse/parse-amount.ts and core/parse/date-engine.ts; write only the Revolut column-name mapping (and a vendor-specific parse-amount/parse-date only if the format is unusual).
  2. execute service — compose the core resolve helpers + a per-row createTransaction loop with onProgress. No re-implemented find-or-create.
  3. queue bundle — one createImportJobQueue call (new base name + a new SSE event type in sse.ts) exposing queueRevolutImport + getRevolutImportProgress.
  4. controllers — parse + status via the shared factories; an execute (and optionally detect-duplicates) controller with a Zod schema reusing the shared mapping schemas.
  5. route file + register — a small revolut.route.ts wired in setup-routes.ts; register the queue's worker teardown in setupIntegrationTests.ts.
  6. e2e through HTTP only (per project rules): happy path, empty state, at least one error case.
Net new work One parse file + a thin execute composition + a route file. The field parsers, dedup, resolve helpers, queue scaffold, and the parse/status controllers are all reused. Provider-specific logic (a weird format, a transfer model) is added in the provider folder, never forced into the shared core.

7. Notes & decisions

The body above describes the current logic. These notes record why a few non-obvious choices were made and which alternatives were rejected.

Execute is uniformly queued + re-parsed Every provider's execute runs as a BullMQ job and re-parses the file server-side from fileContent + mappings + skip indices. CSV is not small (it caps at 50 000 rows), so a synchronous write loop inside one HTTP request was a real timeout risk, and posting back up to 50k pre-parsed rows was a multi-MB body. Re-parsing is deterministic, so skipDuplicateIndices still align against the worker's fresh parse. One execution path for all three providers.
detect-duplicates & extract-unique-values stay synchronous These are also full-file stages, but they are interactive request→response steps that return the review list the user acts on, not a fire-and-forget write — a job would add a blank-wait UX and lifecycle complexity. They are read-only and the duplicate query is capped at 10k. The queue scaffold can promote either to a result-producing job later (job → poll for result) if profiling ever shows a timeout, with no rework.
Tag-creation failures are fail-fast All providers create each unique tag/label once, up front (before the row loop), via the shared create-tags-if-needed helper. The helper propagates a creation failure, and it runs outside the per-row try/catch — so a single failed tag aborts the whole import (the job ends failed) rather than silently landing transactions without that tag. This is an up-front batch step, not a per-row failure. Revisit if a provider ever needs best-effort tagging (add a configurable error mode to the helper).
Rejected — dynamic dispatch & a provider registry A single /import/:provider/:stage route resolving the provider from a registry was considered and rejected as overengineering at three providers. It would not remove the genuinely-different per-provider execute/detect schemas (the dispatcher would branch on provider anyway), it loses per-endpoint static typing (req.body becomes a union + cast), it complicates per-stage middleware (the rate-limited stages vs the unlimited status poll), and the provider×stage grid is ragged (CSV has extract-unique-values; YNAB has no detect-duplicates). Per-provider URLs + the two controller factories capture the real duplication without any of that. An ImportProvider interface + registry object were dropped for the same reason — with per-provider routes, nothing would consume them.
Rejected — a declarative "importer = data" engine The verified divergence (Wallet transfer pairing, YNAB two-level categories + starting-balance, out-of-wallet legs) would all live in a customParse escape hatch, so the engine would be declarative only for plain bank CSVs and a thin shim for everything hard. Composing core/ functions gets the same reuse without config-as-data indirection. Likewise: the dedup key (accountId:day:absAmount:type) stays a function, not data; parse stays a provider-owned pure function; there is no base class.
Persisted provider id The provider id is stored on every imported transaction at externalData.importDetails.source via the ImportSource enum (csv, budget-bakers-wallet, ynab, statement-parser). Renaming a provider id therefore requires a data migration to re-tag existing rows, not just a code change.

Risks to keep in mind

  • Blast radius. A bug in core/duplicates/find-duplicates.ts now affects all providers at once. find-duplicates.unit.ts is the gate.
  • FX-date. Mis-wiring the create-accounts-if-needed callback skews refInitialBalance (CSV = today, Wallet/YNAB = earliest date). Each provider wires it explicitly, gated by its e2e.
  • Transfers don't fit ParsedTransactionRow (no source/dest pair). They stay a provider-private concern inside execute; never forced into the shared row type.
